## 🎯 项目概述

一个基于 AI 的智能医院推荐系统，帮助北京用户根据症状找到合适的医院和科室。

**核心功能：**
- 用户输入症状 → DeepSeek AI 分析 → 推荐科室 → 展示权威医院
- 支持多科室推荐（首选 + 备选）
- 77家北京三甲医院数据
- 移动端优先设计（Gemini 风格）

**技术栈：**
- 前端：纯静态 HTML + Tailwind CSS + JavaScript
- AI：DeepSeek API（症状分析）
- 部署：腾讯云 CloudBase（静态托管 + 云函数）
- 代码托管：GitHub

## 🐛 今天解决的关键 Bug

### Bug #1：科室筛选失败 - 最诡异的 Bug

**现象：**
- 搜索"嗓子疼发烧"，AI 推荐"呼吸内科"
- 应该显示6家医院（中日友好、协和、北大一院等）
- 实际只显示2家医院（同仁、儿童医院）

**排查过程：**
1. ✅ 检查数据 - 中日友好医院数据正确
2. ✅ 检查筛选逻辑 - 代码逻辑正确
3. ✅ 控制台手动测试 - 能正确筛选出6家医院
4. ❌ 实际运行 - 只显示2家医院

**真正原因：**
```javascript
// AI 返回的科室名称带了 Markdown 格式
deptName = "**呼吸内科**"  // 不是 "呼吸内科"

// 导致字符串方法失效
"**呼吸内科**".endsWith('内科')  // false（实际是 endsWith('**')）
```

**关键发现：**
- 用 `Array.from(str).map(c => c.charCodeAt(0))` 查看字符码
- 发现前后各有两个 `42`（星号 `**`）
- AI 输出了 Markdown 加粗格式！

**解决方案：**
```javascript
// 在云函数中清理 Markdown 格式符号
department = department.replace(/[✅🏥💡🔄•\s*_`【】]/g, '');

// 在 prompt 中明确要求
"重要：科室名称不要加任何格式符号（不要加粗、不要星号）"
```

**经验教训：**
1. **AI 输出格式要严格控制** - 在 prompt 中明确禁止格式符号
2. **调试要看字符码** - 当字符串看起来一样但行为不同时
3. **分层排查问题** - 数据层 → 逻辑层 → 展示层
4. **永远不要相信"看起来一样"** - 隐藏字符是真实存在的

## 📊 数据扩充策略演进

### 初始方案：静态数据文件
- 30家医院，手工录入
- 问题：数据不全，用户搜不到

### 方案A：实时联网搜索（讨论后否决）
```
用户输入 → AI推荐科室 → 联网搜索医院 → 展示
```

**为什么不可行：**
- ❌ 搜索结果质量不可控（可能搜到莆田系、广告）
- ❌ 响应速度慢（5-9秒 vs 当前3秒）
- ❌ 成本高（每次查询都要调用搜索API）
- ❌ 数据结构不统一（需要额外解析）
- ❌ 医疗推荐责任重大（无法保证数据准确性）

**产品决策：**
> "用户来这里是为了找'对症的好医院'，不是找'最近的医院'。如果要找最近的，直接用高德/百度地图更方便。"

### 最终方案：静态数据库 + 智能降级 + 持续补充

**三层筛选逻辑：**
```javascript
// 第1层：有该科室的医院（按排名排序）
const withDept = hospitals.filter(h => 
    h.topDepts.some(d => d.name.includes(deptName))
).sort((a, b) => rankA - rankB);

// 第2层：关键词匹配但没有该科室的医院
const withKeyword = hospitals.filter(h => 
    symptom && h.keywords.some(k => symptom.includes(k))
);

// 第3层：综合医院降级（针对常见科室）
const commonDepts = ['呼吸内科', '皮肤科', '消化内科', ...];
if (commonDepts.includes(deptName)) {
    withGeneral = hospitals.filter(h => h.type === '综合医院')
        .slice(0, 10);
}
```

**数据补充策略：**
- **每天补充3个科室** - 10天覆盖30个常见科室
- **基于权威数据** - 复旦大学医院管理研究所排行榜
- **人工审核** - 确保数据准确性
- **增量更新** - 不重复录入

## 💡 重要产品决策

### 1. 距离功能 - 暂不做

**用户需求分析：**
- 用户来这里是为了找"对症的好医院"，不是找"最近的医院"
- 显示距离会让用户优先考虑"近"，而不是"专业"
- 违背了"推荐优质医院"的初衷

**替代方案：**
- 在医院卡片上显示地址
- 用户点击地址，跳转到地图 APP
- 让专业的地图 APP 做导航

### 2. 反馈按钮 - 改为收藏功能

**原方案：** 卡片上加"准确/不准"按钮

**问题分析：**
- 用户还没去医院，怎么知道准不准？
- "不准"的含义模糊（科室不对？医院不擅长？）
- 负面反馈会打击用户信心

**更好的方案：**
- **收藏功能**（正向操作，不打击信心）
- **就诊后反馈**（7天后推送问卷）

### 3. 数据管理 - 静态数据库 + 云数据库

**短期（50-100家）：** 静态 data.js 文件
- 简单、快速
- 适合 MVP 阶段

**长期（100+家）：** 腾讯云数据库
- 数据和代码分离
- 支持增量更新
- 可以在线管理

## 🎓 技术经验总结

### 1. AI 输出格式控制

**问题：** AI 返回的数据带了 Markdown 格式（`**加粗**`）

**解决方案：**
```javascript
// 方法1：在 prompt 中明确要求
"重要：科室名称不要加任何格式符号（不要加粗、不要星号）"

// 方法2：在解析时清理格式符号
department = department.replace(/[✅🏥💡🔄•\s*_`【】]/g, '');
```

### 2. 字符串调试技巧

**当字符串看起来一样，但行为不同时：**
```javascript
// 查看字符码
Array.from(str).map(c => c.charCodeAt(0))
// 输出：[42, 42, 21628, 21560, 20869, 31185, 42, 42]
// 发现前后各有两个 42（星号）
```

### 3. 正则表达式的坑

**问题：** `"呼吸内科".replace(/(内科|外科|学科)$/, '')` 在某些环境下不工作

**解决方案：** 用简单的字符串方法替代
```javascript
// 不用正则
let coreKeyword = deptName;
if (deptName.endsWith('内科')) {
    coreKeyword = deptName.slice(0, -2);
}
```

### 4. 腾讯云 CloudBase 部署经验

**问题汇总：**
- 云函数默认超时3秒 → 改为15秒
- 自动部署不稳定 → 手动上传文件
- 浏览器缓存顽固 → 测试时用无痕模式
- HTTP trigger 参数解析 → 需要 `JSON.parse(event.body)`

### 5. 数据结构设计

**医院数据结构：**
```javascript
{
    name: "北京协和医院",
    rank: "三甲",
    type: "综合医院",
    founded: "1921年",
    desc: "...",
    topDepts: [
        { name: "病理科", rank: 1 },
        { name: "风湿免疫科", rank: 1 },
        ...
    ],
    features: ["罕见病诊疗", "多学科协作"],
    address: "东城区帅府园1号",
    phone: "010-69156114",
    keywords: ["疑难杂症", "罕见病", ...]
}
```

**设计要点：**
- `topDepts` 只录入优势科室（全国前10）
- `keywords` 用于症状匹配
- `features` 用于标签展示
- `rank` 用于排序

## 💬 重要对话摘录

### 关于距离功能

**二月：** "我们的医院要不要加上距离，显示这些医院目前距离她多远？"

**教员分析：**
> 作为产品经理，我的判断：暂不做。
> 
> **理由：**
> - 不符合当前核心价值 - 用户来这里是为了找"对症的好医院"，不是找"最近的医院"
> - 可能误导用户 - 显示距离会让用户优先考虑"近"，而不是"专业"
> - 技术成本高，收益低 - 需要获取位置权限、调用地图API、处理定位失败
> 
> **更好的替代方案：**
> 在医院卡片上显示地址，用户点击地址跳转到地图 APP，让专业的地图 APP 做导航。

### 关于反馈按钮

**二月：** "卡片上加上直接反馈？两个按钮：准确、不准？"

**教员分析：**
> 作为 UX 专家，我的分析：
> 
> **"准确/不准"的问题：**
> 1. 用户还没去医院，怎么知道准不准？
> 2. "不准"的含义模糊（是科室不对？医院不擅长？排名不对？）
> 3. 负面反馈会打击用户信心
> 
> **更好的方案：轻量级收藏 + 延迟反馈**
> - 收藏功能（正向操作，不打击信心）
> - 就诊后反馈（7天后推送问卷）

### 关于数据管理

**二月：** "这么多的医院数据，我到底应该什么样的形式给你？是不是放到云端给你 API 更适合？"

**教员分析：**
> **短期（50-100家）：** 静态 JSON 文件
> - 简单、快速，适合 MVP 阶段
> 
> **长期（100+家）：** 腾讯云数据库
> - 数据和代码分离
> - 支持增量更新（不重复写入）
> - 可以在线管理
> 
> **你给我数据的方式：**
> - Excel/CSV 表格（最方便）
> - JSON 文件（结构化）
> - Markdown 文档（可读性好）
> 
> **我的处理流程：**
> 你的数据 → 我转换为 JSON → 批量导入云数据库 → 前端调用 API

### 关于实时搜索方案

**二月：** "这些医院数据你直接能搜到，那我们是否要继续添加数据库呢，是不是用户输入后 AI 推荐出匹配科室，然后这些科室我们联网直接搜出来？"

**教员分析：**
> **实时搜索方案的致命问题：**
> 
> 1. **搜索结果质量不可控** ⚠️⚠️⚠️
>    - 搜索引擎返回的是"网页"，不是"结构化医院数据"
>    - 可能搜到广告、莆田系医院、民营医院
>    - 排序不准确（SEO优化好的医院排前面）
> 
> 2. **响应速度慢** - 5-9秒 vs 当前3秒
> 3. **成本高** - 每次查询都要调用搜索API
> 4. **无法保证医疗安全** - 可能误导用户去不正规的医院
> 
> **最终方案：静态数据库 + 持续补充**
> - 每天补充3个科室
> - 10天覆盖30个常见科室
> - 基于复旦排行榜 + 人工审核

### 关于数据补充方式

**二月：** "我们可以每天向你刚才那样拿出3个科室，进行联网搜索，然后补充数据库。"

**教员：** "完美！每天补充3个科室，10天就能覆盖30个常见科室！"

**二月：** "你这些是联网搜索吗？其实可以让一个子 agent，用千问这种模型去干。"

**实验结果：**
- ❌ 千问模型联网搜索不够精准
- ❌ 2分钟超时对复杂搜索任务不够
- ✅ 改用 Claude 的医学知识库（基于复旦排行榜）更快更准

**最终策略：**
> "我们可以每天向你刚才那样拿出3个科室，进行联网搜索，然后补充数据库，其实没有几天就补充完了。"
