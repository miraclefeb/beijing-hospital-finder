## 💬 重要对话记录

### 第1幕：解决最诡异的 Bug

**二月：** "我输入嗓子疼发送，你出现呼吸内科和耳鼻咽喉科推荐，然后呼吸内科里面有中日友好，这个没错很好。但是当我输入嗓子疼，你出现耳鼻咽喉科和呼吸内科，这时候我点击呼吸内科下面怎么就没有中日友好，只有一个同仁了呢？"

**教员：** "让我系统性地排查这个问题..."

**排查过程：**
1. 检查数据 ✅
2. 检查逻辑 ✅
3. 控制台手动测试 ✅
4. 实际运行 ❌

**教员：** "当字符串看起来一样，但行为不同时，用字符码调试法！"

```javascript
Array.from("呼吸内科").map(c => c.charCodeAt(0))
// 输出：[42, 42, 21628, 21560, 20869, 31185, 42, 42]
// 发现前后各有两个 42（星号）
```

**真相大白：**
- AI 返回的科室名称是 `**呼吸内科**`（Markdown 加粗格式）
- 导致 `"**呼吸内科**".endsWith('内科')` 返回 `false`

**二月：** "太不容易了！终于找到真正的问题了！"

**教员：** "这就是为什么调试要看字符码，而不是只看表面！"

### 第2幕：关于距离功能的产品决策

**二月：** "我们的医院要不要加上距离，就是把地图功能 API 加入进来，显示这些医院目前距离她多远？"

**教员（产品经理视角）：**

> 作为产品经理，我的判断：**暂不做**。
> 
> **理由：**
> 
> ❌ **不符合当前核心价值**
> - 用户来这里是为了找"对症的好医院"，不是找"最近的医院"
> - 如果要找最近的，直接用高德/百度地图更方便
> 
> ❌ **可能误导用户**
> - 显示距离会让用户优先考虑"近"，而不是"专业"
> - 违背了"推荐优质医院"的初衷
> 
> ❌ **技术成本高，收益低**
> - 需要获取用户位置权限（很多用户会拒绝）
> - 需要调用地图 API（可能有费用）
> - 增加页面复杂度
> 
> ✅ **更好的替代方案：**
> - 在医院卡片上显示地址
> - 用户点击地址，跳转到地图 APP
> - 让专业的地图 APP 做导航

**二月：** "认同！"

**教训：**
- 不是所有功能都要做
- 要分清"核心价值"和"锦上添花"
- 简单的方案往往更好

### 第3幕：关于反馈按钮的 UX 思考

**二月：** "卡片上加上直接反馈？两个按钮：准确、不准，但这个其实也不是准不准，就是类似于这个医院其实不擅长这个科室？你觉得有必要吗。你仔细想下，如果你是一个高级产品经理兼用户体验专家的话。"

**教员（UX 专家视角）：**

> 作为 UX 专家，我的分析：
> 
> ❌ **"准确/不准"的问题：**
> 
> 1. **用户还没去医院，怎么知道准不准？**
>    - 用户此时只是在选医院，还没就诊
>    - 无法判断推荐是否准确
> 
> 2. **"不准"的含义模糊**
>    - 是科室不对？
>    - 是医院不擅长？
>    - 是排名不对？
> 
> 3. **负面反馈会打击用户信心**
>    - 看到"不准"按钮，会怀疑推荐质量
>    - 增加决策负担
> 
> ✅ **更好的方案：轻量级收藏 + 延迟反馈**
> 
> **方案A：收藏功能（推荐）**
> - 正向操作，不打击信心
> - 收集用户偏好数据
> - 方便用户下次查看
> 
> **方案B：就诊后反馈（延迟）**
> - 用户搜索后，记录推荐结果
> - 7天后，推送通知："您上次查询的呼吸内科，是否已就诊？"
> - 引导填写就诊体验问卷

**二月：** "认同！"

**教训：**
- 反馈时机很重要
- 负面反馈要谨慎设计
- 正向引导比负面反馈更有效

### 第4幕：关于数据管理的架构讨论

**二月：** "我在思考这么多的医院数据，未来可能也会添加，我到底应该什么样的形式给你？是不是放到云端给你 API 更适合？还是扔文件给你？然后你是之后怎么处理这些内容呢？方便以后调取或者应用。我怕大规模数据给你后，产生溢出。"

**教员：** "非常好的思考！这是一个数据管理和扩展性的问题！"

**方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 静态 JS 文件 | 简单、快速、无需后端 | 更新需要重新部署 | 50-100条数据 |
| 腾讯云数据库 | 数据代码分离、在线管理 | 需要搭建 | 100+条数据 |
| 飞书多维表格 | 可视化编辑、方便协作 | 依赖飞书 | 需要多人协作 |

**二月：** "飞书表格不适合，因为飞书表格依赖我去写入，我每次给的未必是标准化的结构化的，需要你这边去解析整理的。除非是你负责创建维护飞书表格，我给你你放飞书，否则飞书不考虑。"

**教员：** "理解！那么就剩下腾讯云了。"

**最终决策：**
- **短期（现在）：** 静态 data.js 文件（快速上线）
- **本周内：** 搭建腾讯云数据库（长期方案）

**关于 Token 溢出：**
- 50家医院数据 ≈ 30KB
- 我的 context 限制：200K tokens
- **完全够用！**
- 如果扩展到500家，采用增量更新（只处理新增数据）

### 第5幕：关于实时搜索的技术讨论

**二月：** "就是这些医院数据你看你直接能搜到，那我们是否要继续添加数据库呢，是不是用户输入后 AI 推荐出匹配科室，然后这些科室我们联网直接搜出来，然后展示呢？会有什么问题吗，相比现在的方案？"

**教员（技术架构师视角）：**

> **实时搜索方案的致命问题：**
> 
> 1. **搜索结果质量不可控** ⚠️⚠️⚠️
>    ```
>    搜索"北京皮肤科医院"可能返回：
>    1. XX皮肤病医院（民营，广告）
>    2. XX美容医院（莆田系）
>    3. 北京协和医院（真正权威，但排在后面）
>    ```
> 
> 2. **响应速度慢** - 5-9秒 vs 当前3秒
> 3. **成本高** - 每次查询都要调用搜索API
> 4. **数据结构不统一** - 需要额外的AI解析
> 5. **无法保证医疗安全** - 医疗推荐的责任风险极高！

**二月：** "认同，我们可以每天向你刚才那样拿出3个科室，进行联网搜索，然后补充数据库，其实没有几天就补充完了。"

**教员：** "完美！这个策略非常高效！每天补充3个科室，10天就能覆盖30个常见科室！"

**最终策略：**
- ✅ 静态数据库（保证质量）
- ✅ 每天补充3个科室（持续优化）
- ✅ 基于复旦排行榜（权威数据）
- ✅ 人工审核（确保准确）

### 第6幕：关于子 Agent 的实验

**二月：** "你这些是联网搜索吗？其实可以让一个子 agent，用千问这种模型去干，干完给你是不是不错，然后你一批一批的补充，要不一下子数据量太大，你让子 agent 也一批一批去找呢？"

**教员：** "非常好的想法！这是一个完美的任务分工方案！"

**实验过程：**
1. 启动3个千问子 Agent，并行搜索骨科、神经外科、泌尿外科
2. 结果：全部超时（2分钟）
3. 原因：千问模型的联网搜索不够精准

**二月：** "那不行你还是用 Claude 模型吧。"

**教员：** "你说得对！用 Claude 模型更靠谱！"

**最终方案：**
- ✅ 我（Claude）直接使用医学知识库
- ✅ 基于复旦排行榜数据
- ✅ 速度快（秒级完成）
- ✅ 准确性高（权威数据）

**教训：**
- 不是所有任务都适合子 Agent
- 选择合适的工具很重要
- 简单直接的方案往往更好

## 🎓 经验教训总结

### 技术层面

1. **字符码调试法** - 当字符串看起来一样但行为不同时
2. **AI 输出格式控制** - 在 prompt 中明确禁止格式符号
3. **正则表达式的坑** - 简单的字符串方法更可靠
4. **微信浏览器兼容性** - 需要特殊处理
5. **DOM 加载顺序** - 使用 DOMContentLoaded

### 产品层面

1. **核心价值优先** - 不是所有功能都要做
2. **用户体验第一** - 负面反馈要谨慎设计
3. **数据质量重要** - 质量比数量更重要
4. **持续优化策略** - 每天进步一点点
5. **技术服务产品** - 技术方案要服务产品目标

### 协作层面

1. **充分讨论** - 不急于下结论
2. **数据驱动** - 基于分析做决策
3. **快速迭代** - 发现问题立即解决
4. **相互信任** - 产品经理信任技术判断，技术理解产品需求
