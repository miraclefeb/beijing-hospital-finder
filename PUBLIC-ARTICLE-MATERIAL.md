## 🐛 今天解决的最诡异 Bug

### 问题现象

**用户反馈：**
> "我输入'嗓子疼发烧'，AI 推荐'呼吸内科'，能看到中日友好医院。但我输入'嗓子疼'，AI 也推荐'呼吸内科'，却只显示同仁医院，中日友好医院不见了！"

**技术表现：**
- 数据正确 ✅
- 逻辑正确 ✅
- 控制台手动测试正确 ✅
- 实际运行错误 ❌

### 排查过程

**第1步：检查数据**
```javascript
// 中日友好医院的数据
topDepts: [
    { name: "呼吸内科", rank: 2 }
]
// ✅ 数据正确
```

**第2步：检查逻辑**
```javascript
// 筛选逻辑
const withDept = hospitals.filter(h => 
    h.topDepts.some(d => d.name.includes("呼吸内科"))
);
// ✅ 逻辑正确
```

**第3步：控制台手动测试**
```javascript
filterHospitalsByDept("呼吸内科");
// ✅ 能正确筛选出6家医院
```

**第4步：实际运行**
```javascript
// 页面只显示2家医院
// ❌ 为什么？
```

### 关键突破：字符码调试法

**当字符串"看起来一样"，但行为不同时：**

```javascript
const deptName = "呼吸内科"; // 从 AI 返回的
console.log(deptName); // 显示：呼吸内科
console.log(deptName.endsWith('内科')); // 返回：false ❓

// 用字符码查看真相
Array.from(deptName).map(c => c.charCodeAt(0));
// 输出：[42, 42, 21628, 21560, 20869, 31185, 42, 42]
//       **   **   呼     吸     内     科     **   **
```

**真相大白：**
- AI 返回的科室名称是 `**呼吸内科**`（Markdown 加粗格式）
- 所以 `"**呼吸内科**".endsWith('内科')` 返回 `false`
- 实际是 `endsWith('**')`

### 解决方案

**方法1：在云函数中清理格式符号**
```javascript
// 清理所有可能的格式符号
department = department.replace(/[✅🏥💡🔄•\s*_`【】]/g, '');
```

**方法2：在 prompt 中明确要求**
```javascript
const prompt = `...
重要：科室名称不要加任何格式符号（不要加粗、不要星号）
`;
```

### 经验教训

1. **永远不要相信"看起来一样"** - 隐藏字符是真实存在的
2. **字符码是终极真相** - `charCodeAt()` 不会骗人
3. **AI 输出要严格控制** - 在 prompt 中明确格式要求
4. **多层防御** - prompt 控制 + 代码清理

## 💡 重要的产品决策

### 决策1：距离功能 - 暂不做

**讨论：**
> "我们的医院要不要加上距离，显示这些医院目前距离她多远？"

**分析：**

**用户需求本质：**
- 用户来这里是为了找"对症的好医院"
- 不是找"最近的医院"
- 如果要找最近的，直接用地图 APP 更方便

**可能的问题：**
- 显示距离会让用户优先考虑"近"，而不是"专业"
- 违背了"推荐优质医院"的初衷
- 需要获取位置权限（很多用户会拒绝）
- 技术成本高，收益低

**更好的方案：**
- 在医院卡片上显示地址
- 用户点击地址，跳转到地图 APP
- 让专业的地图 APP 做导航

**决策：暂不做距离功能**

---

### 决策2：反馈按钮 - 改为收藏功能

**讨论：**
> "卡片上加上直接反馈？两个按钮：准确、不准？"

**分析：**

**"准确/不准"的问题：**
1. 用户还没去医院，怎么知道准不准？
2. "不准"的含义模糊（科室不对？医院不擅长？）
3. 负面反馈会打击用户信心

**更好的方案：**
- **收藏功能** - 正向操作，不打击信心
- **就诊后反馈** - 7天后推送问卷，引导填写体验

**决策：不做"准确/不准"按钮，改为收藏功能**

---

### 决策3：数据管理 - 静态数据库 + 云数据库

**讨论：**
> "这么多的医院数据，我到底应该什么样的形式给你？是不是放到云端给你 API 更适合？"

**分析：**

**方案对比：**
- **静态文件** - 简单快速，适合 MVP（50-100条）
- **云数据库** - 数据代码分离，适合长期（100+条）
- **飞书表格** - 可视化编辑，但需要标准化输入

**Token 溢出问题：**
- 50家医院 ≈ 30KB
- Context 限制：200K tokens
- **完全够用！**
- 如果扩展到500家，采用增量更新

**决策：短期用静态文件，本周内搭建云数据库**

---

### 决策4：实时搜索 - 否决

**讨论：**
> "这些医院数据你直接能搜到，那我们是否联网直接搜出来展示呢？"

**分析：**

**实时搜索的致命问题：**
1. **质量不可控** - 可能搜到莆田系、广告
2. **响应速度慢** - 5-9秒 vs 当前3秒
3. **成本高** - 每次查询都要调用 API
4. **医疗安全** - 无法保证数据准确性

**更好的方案：**
- 静态数据库（保证质量）
- 每天补充3个科室（持续优化）
- 基于复旦排行榜（权威数据）

**决策：否决实时搜索，坚持静态数据库 + 持续补充**

## 🎓 技术亮点

### 1. 三层筛选逻辑

```javascript
// 第1层：有该科室的医院（按排名排序）
const withDept = hospitals.filter(h => 
    h.topDepts.some(d => d.name.includes(deptName))
).sort((a, b) => rankA - rankB);

// 第2层：关键词匹配但没有该科室的医院
const withKeyword = hospitals.filter(h => 
    symptom && h.keywords.some(k => symptom.includes(k))
);

// 第3层：综合医院降级（针对常见科室）
if (commonDepts.includes(deptName)) {
    withGeneral = hospitals.filter(h => h.type === '综合医院')
        .slice(0, 10);
}

return [...withDept, ...withKeyword, ...withGeneral];
```

**优势：**
- 优先显示专科医院
- 降低"搜不到"的挫败感
- 保证用户总能看到结果

### 2. AI Prompt 优化

**简洁高效的 Prompt：**
```
你是智能导诊助手。症状：${symptom}

输出格式（不要#号）：

💡 导诊建议
[一句话共情+结论]

🏥 挂号策略
✅ 首选科室：[科室名]
[一句话解释]

🔄 备选科室：
• [科室A]：[简短理由]
• [科室B]：[简短理由]

重要：科室名称不要加任何格式符号（不要加粗、不要星号）
```

**优势：**
- Token 消耗少（max_tokens: 250）
- 格式统一，易于解析
- 明确禁止格式符号

---

### 3. 数据结构设计

```javascript
{
    name: "北京协和医院",
    rank: "三甲",
    type: "综合医院",
    topDepts: [
        { name: "病理科", rank: 1 },
        { name: "风湿免疫科", rank: 1 },
        ...
    ],
    features: ["罕见病诊疗", "多学科协作"],
    keywords: ["疑难杂症", "罕见病", ...]
}
```

**设计要点：**
- `topDepts` 只录入优势科室（全国前10）
- `keywords` 用于症状匹配
- `features` 自动生成标签
- 结构清晰，易于扩展
